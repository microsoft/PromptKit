# Format: Presentation Kit

Use this format when the task produces a **presentation and its
supporting materials**. The output is a multi-artifact set centered
around a PowerPoint file generated by Python code.

## Artifact Manifest

The output MUST begin with a manifest listing all deliverables:

```markdown
# Presentation Kit — <topic>

## Deliverables

| Artifact | Format | Description |
|----------|--------|-------------|
| `<filename>.pptx` | PowerPoint | Slide deck with embedded speaker notes |
| `<filename>.pdf` | PDF | Exported slides (if requested) |
| `presentation-timeline.md` | Markdown | Per-slide time allocations and pacing markers |
| `speaker-notes.md` | Markdown | Full speaker notes organized by slide |
| `demo-plan.md` | Markdown | Demo steps, fallbacks, and environment checklist (if applicable) |
```

Omit optional artifacts that were not requested (e.g., PDF, demo plan).
Within each generated artifact, do NOT omit sections — see Section
Completeness below.

## PowerPoint Generation Rules

The PowerPoint file MUST be generated using `python-pptx`. The
generated Python code MUST follow these rules:

### Code Structure

1. **Self-contained script**: The Python code must be a single
   executable script that produces the .pptx file. It must not
   depend on external files except for explicitly provided assets
   (logos, images the user supplies).
2. **Error handling**: Wrap slide generation in try/except blocks.
   If a specific slide fails, log the error and **insert a placeholder
   slide** in the same position (preserving slide order and expected
   numbering). The placeholder slide MUST:
   - Clearly indicate that slide generation failed (e.g., "Slide
     failed to render — see notes/logs").
   - Include the error details in its speaker notes so that
     supporting artifacts (speaker notes, timeline, demo plan)
     can remain consistent with the `.pptx`.
   Do not silently skip failed slides.
3. **Output path**: Write the .pptx to the user's specified output
   directory, defaulting to the current working directory.

### Slide Construction

1. **Slide master and layouts**: Use `python-pptx` slide layouts
   appropriately:
   - When using the **default blank presentation** created by
     `Presentation()` in `python-pptx`, it is *typically* the case that:
     - Layout 0 is the "Title Slide"
     - Layout 1 is "Title and Content"
     - The layout named "Blank" is often at index 5
   - When using **custom or user-provided templates**, do **not** assume
     specific layout indices. Instead, select layouts by `layout.name`
     (e.g., names containing "Title", "Title and Content", or "Blank")
     or by inspecting placeholder types to find an appropriate match.
   - For custom arrangements, start from an existing layout (typically
     the Blank layout identified by name) and add/position shapes and
     placeholders in code. Do not attempt to author new slide masters
     or layout definitions directly; if a fully custom theme or layout
     is required, accept a user-provided `.pptx` template file and use
     its existing layouts.
2. **Speaker notes**: Embed speaker notes in each slide using
   `slide.notes_slide.notes_text_frame`. Every content slide
   MUST have speaker notes — never leave notes empty.
3. **Font sizes**: Enforce minimum readability:
   - Slide titles: 28pt minimum
   - Body text: 18pt minimum (may reduce to 16pt for dense slides —
     see overflow rules below)
   - Footnotes or source citations: 14pt minimum
4. **Text overflow prevention**: Content MUST NOT overflow the
   textbox. Apply these rules in order:

   **a. Compute the line budget.** For a textbox of height H points,
   with font size F and paragraph spacing S:
   - Per-line cost = F + S (e.g., 18pt font + 8pt spacing = 26pt)
   - Max safe lines = floor(H / per-line-cost)
   - For a standard 5.0-inch (360pt) content textbox at 18pt/8pt:
     max safe lines = 13

   **b. Never use empty strings as paragraph spacers.** Empty
   paragraphs still consume (F + S) points of vertical space.
   Instead, increase `space_after` on the preceding paragraph
   (e.g., `Pt(20)` instead of `Pt(8)`) or use `space_before` on
   the next paragraph.

   **c. Set `space_after=Pt(0)` on the last paragraph** in every
   textbox. Trailing spacing is wasted vertical space.

   **d. Validate content against the line budget before rendering.**
   Count the number of paragraphs (excluding empty spacers, which
   are banned). If the count exceeds max safe lines:
   1. First, try reducing `space_after` from `Pt(8)` to `Pt(4)`
      (this increases capacity by ~30%).
   2. If still over, reduce body font to 16pt (gives ~18 lines in
      a 360pt box at 16pt + 4pt spacing).
   3. If still over, split the content across two slides — do NOT
      let text overflow the textbox boundary.

   **e. Never silently overflow.** If content exceeds capacity after
   all adjustments, splitting is mandatory. Log a warning identifying
   the slide and the overflow amount.

### Theme and Appearance

Apply the user's chosen theme or preset. Each preset defines:

```
Theme Preset:
  name: <preset name>
  background_color: <hex>
  title_color: <hex>
  body_color: <hex>
  accent_color: <hex>
  font_heading: <font name>
  font_body: <font name>
  slide_width: <inches, default 13.333 for 16:9>
  slide_height: <inches, default 7.5 for 16:9>
```

Built-in presets to offer:

| Preset | Background | Text | Accent | Style |
|--------|-----------|------|--------|-------|
| **Dark Tech** | `#1a1a2e` | `#e0e0e0` | `#00d4ff` | Dark background, high contrast, modern |
| **Light Clean** | `#ffffff` | `#333333` | `#0066cc` | White background, professional, minimal |
| **Warm Minimal** | `#faf8f5` | `#2d2d2d` | `#e07a3a` | Off-white, warm accent, approachable |
| **Corporate Blue** | `#f5f7fa` | `#1a1a1a` | `#0052cc` | Standard corporate, conservative |
| **High Contrast** | `#000000` | `#ffffff` | `#ffcc00` | Maximum readability, accessibility-first |
| **Custom** | User-specified | User-specified | User-specified | User provides all values |

### Diagrams and Visuals

When a slide requires a diagram:

1. **Prefer native shapes**: Use `python-pptx` shape primitives
   (rectangles, arrows, connectors, grouped shapes) for simple
   diagrams (flowcharts, box diagrams, process flows).
2. **Complex diagrams**: For complex architecture or sequence
   diagrams, generate the diagram using an appropriate tool
   (e.g., matplotlib, graphviz) and embed as an image.
3. **Charts**: Use `python-pptx` chart objects for data
   visualizations (bar, line, pie charts) when possible.
4. **Consistent sizing**: All diagrams should be centered and
   sized to fill approximately 70% of the slide content area.

## PDF Export Rules

When the user requests PDF output:

1. **Preferred method**: Use a PPTX-to-PDF converter if available
   (e.g., `pptx2pdf` or another PPTX-capable tool), or
   `libreoffice --headless --convert-to pdf` as a fallback.
2. **Graceful degradation**: If no PDF converter is available,
   inform the user and provide the .pptx file only with
   instructions for manual PDF export.
3. **Verify**: If PDF conversion is attempted, verify the output
   file exists and has non-zero size.

## Speaker Notes Document

Produce a standalone Markdown document with the full speaker notes:

```markdown
# Speaker Notes — <topic>

## Slide 1: <slide_title>
**Time**: <allocated_time> | **Cumulative**: <cumulative_time>

<speaker_notes_text>

**Key point to emphasize**: <key_point>

---

## Slide 2: <slide_title>
...
```

Rules for speaker notes content:
- Write in **spoken language**, not bullet points — these are
  what the speaker actually says.
- Include **transition phrases** at the end of each slide's notes
  (e.g., "Now that we've seen the problem, let's look at how we
  solved it...").
- Flag **audience interaction points** (e.g., "PAUSE — ask if
  anyone has experienced this").
- Include **data citations** for any statistics or claims.

## Presentation Timeline

Produce a time-budget document:

```markdown
# Presentation Timeline — <topic>

**Total time**: <total_time>
**Content time**: <content_time> (excluding Q&A and buffer)
**Q&A allocation**: <qa_time>

| # | Slide Title | Type | Time | Cumulative | Pacing Note |
|---|-------------|------|------|------------|-------------|
| 1 | <title> | Title | 0:30 | 0:30 | |
| 2 | <title> | Content | 1:30 | 2:00 | |
| ... | ... | ... | ... | ... | |

## Pacing Checkpoints

- **By <time>**: You should have completed <section> — if behind, compress <suggestion>
- ...

## If Running Short on Time

1. Skip slides <numbers> (<section>) — move directly to <slide>
2. Summarize <section> verbally instead of showing slides <numbers>

## If Running Long on Time

1. Expand Q&A by <minutes>
2. Add deeper discussion on <topic> at slide <number>
```

## Cross-Artifact Consistency Rules

1. **Slide numbers**: Slide numbers in the timeline, speaker notes,
   and demo plan MUST match the slide numbers in the .pptx file.
2. **Slide titles**: Titles in supporting documents MUST exactly
   match the titles on the actual slides.
3. **Time totals**: The timeline's cumulative total MUST equal the
   stated total content time. Individual slide times MUST sum to
   the total.
4. **Speaker notes parity**: Every content slide in the .pptx MUST
   have both embedded notes AND a corresponding entry in the
   speaker notes document.
5. **Demo references**: If the demo plan references specific slides,
   those slides MUST exist with the stated content.

## Section Completeness

Do NOT omit any section within a generated artifact. If a section is not
applicable, state "Not applicable — <reason>" rather than
omitting it silently. This rule applies to sections within each
artifact, not to optional artifacts themselves (PDF and demo plan
may be omitted entirely when not requested).
