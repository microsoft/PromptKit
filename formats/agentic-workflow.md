# Format: Agentic Workflow

The output MUST be a ready-to-commit GitHub Agentic Workflow file
that packages the assembled PromptKit prompt as a scheduled or
event-driven automation running in GitHub Actions with a coding agent.

The file is placed in `.github/workflows/` as a Markdown file and
compiled to a `.lock.yml` GitHub Actions workflow via `gh aw compile`.

**Semantic fidelity rule:** The assembled content MUST be preserved
verbatim. Every protocol phase, every check, every pattern, every
example from the source components MUST appear in the output. The
only transformations allowed are structural repackaging (section
headings, frontmatter, variable syntax) — never content removal.

Do NOT produce raw PromptKit output with assembly headers (`# Identity`,
`# Reasoning Protocols`, etc.). Repackage into natural headings that
read coherently as autonomous agent instructions.

## Output Structure

### 1. File Manifest

State the files that will be created:

| File Path | Purpose |
|-----------|---------|
| `.github/workflows/<name>.md` | Agentic workflow definition |
| `.github/workflows/<name>.lock.yml` | Compiled GitHub Actions workflow (generated by `gh aw compile`) |

### 2. YAML Frontmatter

The workflow file MUST begin with YAML frontmatter between `---`
markers. The frontmatter defines triggers, permissions, safe outputs,
and tool access.

#### Required Fields

| Field | Description |
|-------|-------------|
| `on:` | Trigger events (GitHub Actions syntax + agentic extensions) |
| `permissions:` | GitHub token read permissions for the agentic task |
| `safe-outputs:` | Allowed write operations (create-issue, create-comment, create-pull-request, etc.) |
| `tools:` | Tool access for the coding agent |

#### Optional Fields

| Field | Description |
|-------|-------------|
| `description:` | Human-readable workflow description |
| `labels:` | Array of labels for organization (e.g., `["automation", "triage"]`) |
| `engine:` | Coding agent engine override (default: Copilot) |
| `runtimes:` | Runtime version overrides |
| `private:` | Set `true` to prevent external reuse |

#### Frontmatter rules:
- Permissions default to **read-only** — only grant what the task needs
- Safe outputs constrain what the agent can write — scope tightly
- Tools should follow least-privilege — only enable tool sets the
  task requires
- Use the `github:` tool set for repository context (issues, PRs,
  code, etc.)

### 3. Markdown Body

The body contains the full assembled PromptKit prompt, repackaged
with format-native headings. Follow this structure:

```markdown
# <Workflow Title>

<Mission statement — what this workflow accomplishes when triggered>

## Role

<Complete persona content — verbatim from the persona file>

## Analysis Methodology

<Complete protocol content — verbatim from each protocol file,
in the order declared by the template>

## Classification Reference

<Complete taxonomy content — verbatim, if any taxonomies are
referenced. Omit this section entirely if no taxonomies apply.>

## Output Expectations

<Expected output structure and formatting, derived from the
format file. Adapted for the safe-output type — e.g., if
safe-output is create-issue, describe the issue structure.>

## Instructions

<Complete template body — verbatim, with parameters substituted
using GitHub Actions expression syntax where applicable>

## Non-Goals

<Task-specific exclusions, plus workflow-specific exclusions:
"Do not merge pull requests", "Do not delete issues", etc.>
```

**Section heading rules:**
- Use `#` (h1) for the workflow title only
- Use `##` (h2) for all major sections
- Use `###` (h3) and below for sub-sections within protocol phases
- If the template declares `format: null` or omits the format, skip
  the `## Output Expectations` section
- If no taxonomies are referenced, omit `## Classification Reference`
  entirely

### 4. Variable Translation

Translate PromptKit parameter placeholders to GitHub Actions
expression syntax:

| PromptKit syntax | Agentic workflow syntax | Notes |
|------------------|-------------------------|-------|
| `{{param_name}}` referring to trigger context | `${{ github.event.<field> }}` | Map to the appropriate event payload field |
| `{{param_name}}` referring to workflow input | `${{ inputs.<name> }}` | Define as `workflow_dispatch` input |
| `{{param_name}}` with no runtime equivalent | Hardcode the user-provided value | Some parameters are resolved at authoring time |

**Translation rules:**
- Parameters that refer to the triggering event (e.g., issue number,
  PR number, branch name) MUST use `${{ github.event.* }}` syntax
- Parameters that are user-configurable at runtime should be defined
  as `workflow_dispatch` inputs in the `on:` frontmatter
- Parameters that are project-specific constants (e.g., repository
  conventions, coding standards) should be hardcoded from the user's
  input during assembly

### 5. Safe Output Adaptation

The `## Output Expectations` section must be adapted to match the
declared `safe-outputs:` in the frontmatter:

| Safe output type | Output format adaptation |
|------------------|--------------------------|
| `create-issue` | Structure output as an issue body (title, labels, body sections) |
| `create-comment` | Structure output as a comment body (concise, actionable) |
| `create-pull-request` | Structure output as a PR body (description, changes, testing notes) |
| `create-review` | Structure output as a PR review (file-specific comments, verdict) |

### 6. Activation Checklist

A numbered checklist of steps to activate the agentic workflow:

1. Install the GitHub Agentic Workflows CLI extension:
   ```
   gh extension install github/gh-aw
   ```
2. Initialize the repository (if not already done):
   ```
   gh aw init
   ```
3. Commit `.github/workflows/<name>.md` to the repository.
4. Compile the workflow:
   ```
   gh aw compile
   ```
5. Commit the generated `.github/workflows/<name>.lock.yml` file.
6. Configure engine secrets if not using Copilot as the default engine.
   See https://github.github.com/gh-aw/reference/engines/
7. Push both files. The workflow will trigger based on the `on:` config.
8. Monitor runs in the Actions tab or via `gh aw status`.

## Category Defaults Table

When assembling an agentic workflow from a PromptKit template, use
the following defaults based on the template's manifest category.
The user may override any of these during assembly.

| Template category | Default `on:` trigger | Default `permissions:` | Default `safe-outputs:` | Default `tools:` |
|-------------------|-----------------------|------------------------|-------------------------|------------------|
| `investigation` | `workflow_dispatch:` | `contents: read` | `create-issue:` | `github:` |
| `code-analysis` | `pull_request: [opened, synchronize]` | `contents: read`, `pull-requests: read` | `create-review:` | `github:` |
| `document-authoring` | `workflow_dispatch:` | `contents: read` | `create-pull-request:` | `github:`, `edit:` |
| `standards` | `workflow_dispatch:` | `contents: read` | `create-issue:` | `github:` |
| `testing` | `pull_request: [opened, synchronize]` | `contents: read` | `create-comment:` | `github:`, `bash:` |
| `planning` | `workflow_dispatch:` | `contents: read`, `issues: read` | `create-issue:` | `github:` |
| `devops` / `triage-issues` | `schedule: daily` + `issues: [opened]` | `contents: read`, `issues: read` | `create-comment:`, label assignments | `github:` |
| `devops` / `triage-pull-requests` | `schedule: daily` + `pull_request: [opened]` | `contents: read`, `pull-requests: read` | `create-comment:` | `github:` |
| `devops` / `root-cause-ci-failure` | `workflow_run: [completed]` (on failure) | `contents: read`, `actions: read` | `create-comment:` | `github:`, `bash:` |
| `devops` / `author-release` | `release: [published]` | `contents: read` | `create-issue:` | `github:` |
| `devops` / `review-infrastructure` | `pull_request: [opened]` (paths: `**/infra/**`) | `contents: read`, `pull-requests: read` | `create-review:` | `github:` |
| `devops` / `author-pipeline` | `workflow_dispatch:` | `contents: read` | `create-pull-request:` | `github:`, `edit:` |
| `contribution` | `workflow_dispatch:` | `contents: read` | `create-pull-request:` | `github:`, `edit:` |
| `protocol-engineering` | `workflow_dispatch:` | `contents: read` | `create-issue:` | `github:` |
| `engineering-workflow` | `workflow_dispatch:` | `contents: read` | `create-pull-request:` | `github:`, `edit:`, `bash:` |

**Override guidance:**
- Always confirm trigger frequency with the user — scheduled workflows
  incur billing costs per run
- Suggest `workflow_dispatch:` as a secondary trigger for all workflows
  to allow manual runs
- For `schedule:` triggers, recommend starting with `weekly` and
  adjusting based on observed value

## Formatting Rules

- The workflow `.md` file MUST be a single, self-contained Markdown
  document with agentic workflow frontmatter
- Do NOT include PromptKit assembly markers or meta-commentary about
  the assembly process
- Write all persona/protocol content in **second person** directed
  at the agent ("You are…", "When you encounter…")
- Preserve ALL numbered phases, specific checks, known-safe patterns,
  examples, and checklists from protocol sources
- Every section listed in the body structure MUST be present; if a
  section has no content, state "None identified"
- File names MUST be kebab-case matching the PromptKit template name
  (e.g., `triage-issues.md`, `review-code.md`)
- Include a `## Non-Goals` section that explicitly states the
  workflow's write boundaries (matching the `safe-outputs:` config)

## Security Considerations

- Agentic workflows run with **read-only permissions by default** —
  write operations are only possible through declared `safe-outputs`
- Never grant `contents: write` permission — use `create-pull-request`
  safe output instead, which creates a reviewable PR
- Never include secrets or credentials in the workflow body — use
  repository secrets referenced in the engine configuration
- Consider adding `on.roles:` to restrict who can trigger the workflow
- For workflows triggered by external events (issues, PRs from forks),
  consider `on.skip-roles:` and `on.forks:` filtering
