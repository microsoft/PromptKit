# Case Study: Generating Agent Instructions with PromptKit

## The Problem

A team wants their GitHub Copilot to consistently follow memory-safety
practices when working on their C codebase. They've tried adding comments
to their code and writing long system prompts, but:

- The prompts aren't reusable across projects
- Each developer writes slightly different instructions
- There's no review process for prompt quality
- The prompts drift over time as people copy-paste-modify

They need persistent, version-controlled agent instructions that load
automatically and enforce consistent behavior.

## The PromptKit Approach

### Assembling the Prompt

```bash
npx promptkit assemble author-agent-instructions \
  -p platform="GitHub Copilot" \
  -p base_persona="systems-engineer" \
  -p selected_protocols="anti-hallucination, memory-safety-c, thread-safety" \
  -p behaviors="Memory-safe C coding, thread-safe data structure access, \
     defensive error handling" \
  -p scope="project" \
  -p context="Linux kernel-style C codebase. Uses pthreads for concurrency. \
     Custom allocator in src/alloc/. No C++ or smart pointers." \
  -o agent-instructions-prompt.md
```

### What Gets Assembled

The prompt composes the `spl-contributor` persona with anti-hallucination
and self-verification protocols, plus the `agent-instructions` format. The
template body instructs the LLM to:

1. **Load** the specified persona and protocols
2. **Plan decomposition** — decide how to split instructions into composable
   skill files
3. **Condense** protocols into standing directives (not step-by-step
   instructions)
4. **Apply platform constraints** — GitHub Copilot's ~4 KB per skill file
   limit, `applyTo` glob targeting
5. **Produce output files** — ready-to-commit `.instructions.md` files
6. **Verify** output against the format spec

### What Gets Produced

When you feed the assembled prompt to an LLM, it generates files like:

**`.github/instructions/memory-safety.instructions.md`**
```yaml
---
description: "Memory safety practices for C code"
applyTo: "**/*.c, **/*.h"
---
```
```markdown
<!-- Generated by PromptKit — edit with care -->

## Memory Safety

When writing or reviewing C code in this project:

- Track allocation/deallocation pairs. Every `malloc` must have a
  corresponding `free` on all code paths, including error paths.
- Validate all pointer arithmetic against buffer bounds before
  dereferencing.
- Use the project's custom allocator (`src/alloc/`) — do not call
  `malloc`/`free` directly.
- Check return values of all allocation functions. Handle `NULL` returns
  explicitly.
- Label ownership: document whether a function takes ownership of a
  pointer or borrows it.
```

**`.github/instructions/thread-safety.instructions.md`**
```yaml
---
description: "Thread safety practices for concurrent code"
applyTo: "**/*.c, **/*.h"
---
```
```markdown
<!-- Generated by PromptKit — edit with care -->

## Thread Safety

When working with concurrent code:

- Identify shared state before modifying any data structure. If it's
  accessed from multiple threads, it needs synchronization.
- Use the project's mutex wrappers, not raw `pthread_mutex_*` calls.
- Lock ordering: always acquire locks in alphabetical order by variable
  name to prevent deadlocks.
- Do not hold locks across blocking I/O operations.
- Mark thread-local data with `__thread` annotation and document why
  it's thread-local.
```

## The Difference

### Without PromptKit

Each developer writes their own Copilot instructions:

> "When writing C code, be careful with memory. Always free what you
> allocate. Don't forget to check for null pointers."

This is vague, inconsistent across the team, and misses critical
practices (ownership semantics, custom allocator, lock ordering).

### With PromptKit

- **Composable skill files** — each `.instructions.md` covers one concern
  with proper `applyTo` targeting
- **Protocol-derived** — practices come from battle-tested protocols
  (memory-safety-c, thread-safety), not ad-hoc recollection
- **Version-controlled** — committed to the repo, reviewed via PR,
  improved over time
- **Consistent** — every developer gets the same Copilot behavior
- **Platform-aware** — respects GitHub Copilot's file size constraints and
  frontmatter requirements

## Multi-Platform Support

The same prompt can generate instructions for multiple platforms:

```bash
# For all platforms at once
npx promptkit assemble author-agent-instructions \
  -p platform="All" \
  ...
```

This produces:
- `.github/instructions/*.instructions.md` for GitHub Copilot
  (composable skill files with `applyTo` targeting)
- `CLAUDE.md` for Claude Code (single combined file)
- `.cursorrules` for Cursor (single combined file)

## Takeaways

- **Protocols become standing directives.** PromptKit's multi-phase
  analysis protocols are condensed into persistent "always do this" rules
  appropriate for agent instruction files.
- **Composability extends to agent instructions.** Instead of one massive
  instruction file, PromptKit produces focused skill files that can be
  independently updated.
- **The team shares behavior, not just code.** Version-controlled agent
  instructions mean the whole team's AI assistant enforces the same
  practices.
- **Platform differences are handled.** GitHub Copilot's composable skill
  model (multiple files with glob targeting) vs. Claude/Cursor's single-file
  model are both supported from the same source components.
