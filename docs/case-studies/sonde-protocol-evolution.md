# Case Study: Evolving a Cryptographic Protocol with PromptKit

> **One protocol migration, 10+ spec documents, 28 modified requirements,
> 16 retired, 8 new** — guided by an 8-phase interactive workflow that
> caught fundamental design errors at two stages: a wrong crypto model
> before specs were written (Phase 1), and a frame ownership error
> during implementation review (Phase 7).

## The Project

[Sonde](https://github.com/alan-jowett/sonde) is a programmable runtime
for distributed sensor nodes — ESP32 microcontrollers running verified
BPF programs, with a gateway distributing code over ESP-NOW radio and a
BLE pairing tool for device onboarding.

The project uses a specification-first methodology with PromptKit-generated
workflows driving the development process. All changes follow a
requirements → design → validation → implementation → audit cycle.

## The Change

**Issue [#495](https://github.com/alan-jowett/sonde/issues/495)**:
Replace the gateway's asymmetric key-pair (Ed25519/X25519/ECDH/HKDF)
pairing encryption with AES-256-GCM using the pairing tool PSK directly.

What started as "simplify pairing crypto" turned out to be a
**protocol-wide migration** — every radio frame (WAKE, COMMAND, APP_DATA,
PEER_REQUEST, PEER_ACK) switched from HMAC-SHA256 integrity to
AES-256-GCM authenticated encryption.

## The PromptKit Approach

### Workflow Selection

The `evolve` workflow was generated by PromptKit with:
- **Persona**: Senior Systems Engineer
- **Reasoning protocols**: Anti-hallucination guardrails, adversarial
  falsification, self-verification, operational constraints
- **Audit protocols**: Traceability audit (D1-D7), code compliance
  (D8-D10), test compliance (D11-D13)
- **Task**: Incremental development workflow (8 phases)

### The 8-Phase Pipeline

```
Phase 1: Requirements Discovery (interactive)
    ↓
Phase 2: Specification Changes
    ↓
Phase 3: Specification Audit (adversarial)
    ↓  ← REVISE: 4 issues found, fixed, re-audited → PASS
Phase 4: User Review
    ↓
Phase 5: Implementation Changes
    ↓
Phase 6: Implementation Audit
    ↓
Phase 7: User Review of Implementation
    ↓
Phase 8: Create Deliverable
```

## Phase 1: Requirements Discovery — Where the Workflow Proved Its Value

The agent's first attempt at Phase 1 produced a requirements change
manifest with **fundamental errors**:

1. **Wrong crypto model**: The report said "AES-GCM is new" — but the
   existing protocol already used AES-256-GCM for payload encryption
   (via ECDH key agreement). The change was about **where the key comes
   from** (PSK directly vs ECDH-derived), not adding AES-GCM.

2. **Scope underestimation**: The initial report treated this as a
   pairing-only change. The user pointed out that replacing HMAC-SHA256
   with AES-GCM affects **every radio frame**, not just pairing.

3. **Wrong "What doesn't change" section**: Listed frame-level HMAC as
   unchanged — but the entire point was to replace it with GCM tags.

### Interactive Correction

The evolve workflow's Phase 1 requires **explicit user confirmation**
before proceeding. This gate caught the errors. The user walked through
each step of the new protocol interactively:

> **User**: "There is no ECDH in Phase 1. Pairing tool sends its PSK to
> the gateway over BLE, no dynamic key exchange."

> **User**: "There is no frame level HMAC. It's only AES-256-GCM."

> **User**: "The PSK is the node identifier. Everything else is just
> meta-data."

Through 6 rounds of interactive Q&A, the correct model emerged:

**One crypto scheme, one frame format, different keys per role:**

```
Header (AAD, cleartext): key_hint(2B) ‖ msg_type(1B) ‖ nonce(8B)
Body: AES-256-GCM ciphertext ‖ 16-byte GCM tag

gcm_nonce = SHA-256(psk)[0..3] ‖ msg_type ‖ frame_nonce[8]
```

| Message | PSK used |
|---------|----------|
| WAKE, COMMAND, APP_DATA | node_psk |
| PEER_REQUEST | phone_psk |
| PEER_ACK | node_psk |

The Phase 1 report was completely rewritten with the correct
understanding before proceeding.

### What Would Have Gone Wrong Without Phase 1

Without the interactive discovery phase, the agent would have:
- Written spec changes for "add AES-GCM to pairing only"
- Left HMAC-SHA256 on all non-pairing frames
- Kept the gateway asymmetric identity
- Produced a half-migrated protocol with two incompatible crypto models

The Phase 1 gate **prevented an entire category of specification error**
from propagating into code.

## Phase 3: Adversarial Specification Audit

The adversarial audit found **4 issues**:

1. **`gateway-design.md` not updated** (HIGH) — The design document
   still described HMAC-SHA256/ECDH/HKDF in 8 sections.
2. **3 test cases referenced retired requirements** (HIGH)
3. **Wrong PSK in test case** (HIGH) — T-1214 said `node_psk` for
   PEER_REQUEST, should be `phone_psk`.
4. **Missing test for GW-1220** (MEDIUM)

All 4 were fixed and the audit re-run → **PASS**.

## Phase 4: User Review — 6 Rounds of PR Review

The specification PR went through **6 rounds** of automated review,
each finding real issues:

| Round | Findings | Key fix |
|-------|----------|---------|
| 1 | 6 | REGISTER_ACK → PHONE_REGISTERED naming |
| 2 | 10 | phone_key_hint/phone_psk in NODE_PROVISION |
| 3 | 4 | PT-0407 wire format, AAD byte count |
| 4 | 5 | DecodedFrame struct, nonce prefix rationale |
| 5 | 7 | GCM nonce includes msg_type (security fix!) |
| 6 | 7 | PHONE_REGISTERED payload consistency |

Round 5 was particularly significant: the reviewer identified that the
GCM nonce construction should include `msg_type` to prevent nonce reuse
between request/response pairs using the same PSK. This was a **real
cryptographic security fix** found by automated spec review.

## Phase 5: Implementation — Feature-Flag Migration

Implementation used an **additive, feature-flag** approach:

```toml
[features]
aes-gcm-codec = ["sonde-protocol/aes-gcm-codec"]
```

- Old HMAC API stays intact (default)
- New AES-GCM API added alongside, gated by feature flag
- Consumers migrated one at a time
- Old API removed only after all consumers migrated

| Crate | PR | What Changed |
|-------|-----|-------------|
| `sonde-protocol` | #608 | `AeadProvider` trait, `encode_frame_aead`/`decode_frame_aead`/`open_frame` |
| `sonde-gateway` | #610 | `GatewayAead` impl, `process_frame_aead`, shared `_core` handlers |
| `sonde-node` | #611 | `NodeAead` impl, AEAD wake cycle + peer request (4 rounds of review) |
| `sonde-pair` | #612 | `encrypt/decrypt_pairing_request_aead`, simplified registration |
| `sonde-modem` | #613 | Feature flag propagation only (transparent bridge) |
| `sonde-e2e` | #614 | 5 AEAD integration tests |

The initial implementation attempt was **break-before-make** — changing
the protocol API without updating consumers, which broke the entire CI.
The user caught this immediately and directed a switch to feature flags.

### Lesson: Break-Before-Make vs Feature Flags

The agent's instinct was to replace the old API. The user said:

> "I was thinking more of a side by side approach. Everything can
> continue using the old one, but with a compile time option to test
> the new one."

This saved significant time — each crate could be migrated and reviewed
independently with the full test suite passing at every step.

## Phase 6: Implementation Audit

The adversarial implementation audit (D8-D10 code compliance) checked
all 9 AEAD source files against 49 spec requirements.

**Verdict: PASS** — zero D8/D9/D10 findings. All frame format, nonce
construction, AAD, PSK assignment, and error handling matched the spec.

Two documented incomplete items were found (PEER_REQUEST PSK and
gateway rejection) that became the Phase 7 discovery — see below.

## Phase 7: User Review — Frame Ownership Discovery

During Phase 7 review, the user identified a **fundamental design error**
in the PEER_REQUEST frame ownership:

The spec (written in Phase 2) said the node uses `phone_psk` to build
the PEER_REQUEST ESP-NOW frame. The user corrected this:

> "The phone (pairing tool) generates an entire ESP-NOW frame,
> including the header/AAD, encrypted payload, and tag. The node just
> forwards it."

This meant:
- The node doesn't need `phone_psk` at all
- The `sonde-pair` crate builds the complete frame
- The node is a pure relay — zero crypto for PEER_REQUEST
- The gateway accepts PEER_REQUEST on the AEAD path (not reject it)

PR #615 fixed this across specs, node, pair, and gateway code, and
also **enabled `aes-gcm-codec` by default** across all crates.

### Lesson: Interactive Gates Catch Design Errors Late

Even after 6 rounds of spec review and a full implementation audit,
the frame ownership error wasn't caught until the user reviewed the
code. The spec was internally consistent but wrong — only domain
knowledge (how BLE provisioning actually works) could catch it.

## Phase 8: Deliverable

Issue #495 closed with a summary of all changes. The PromptKit case
study (this document) updated to reflect the completed migration.

## Key Takeaways

### 1. Interactive Discovery Prevents Catastrophic Misunderstanding

The LLM agent's initial understanding of the crypto change was
fundamentally wrong. Without the evolve workflow's mandatory
interactive Phase 1 gate, the error would have propagated through
spec changes and into code — producing a half-migrated protocol.

### 2. Adversarial Review Finds Real Security Issues

The automated reviewer found a GCM nonce construction issue in
round 5 of the spec review. This was not a style or formatting
issue — it was a genuine cryptographic vulnerability that would
have caused nonce reuse between request/response pairs.

### 3. Feature Flags Enable Incremental Migration

The initial implementation attempt was break-before-make, which
tanked CI. Switching to a feature-flag approach allowed each crate
to be migrated independently with the full test suite passing at
every step.

### 4. Cross-Document Consistency is Hard for LLMs

The spec changes touched 9+ documents describing the same concepts
(frame format, nonce construction, PHONE_REGISTERED payload). The
agent consistently introduced inconsistencies when editing one
document without checking the others. Adding a mandatory
cross-document consistency audit before pushing reduced but did not
eliminate this problem.

### 5. Spec-First Works — Even for Protocol-Wide Changes

The entire frame codec migration was designed, reviewed, and
audited at the spec level before a single line of implementation
code was written. The specification caught issues (nonce reuse,
missing msg_type binding, wrong PSK assignments) that would have
been much harder to find in code.

### 6. Domain Knowledge Cannot Be Automated

The frame ownership error (Phase 7) survived 6 rounds of automated
spec review, a full adversarial audit, and complete implementation.
Only the user — who understands how BLE provisioning physically
works — caught it. LLM agents can check consistency and find
contradictions, but they cannot substitute for domain expertise
about how the system actually operates in the field.
